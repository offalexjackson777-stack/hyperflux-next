# OpenRazer Compatibility Boundary

The OpenRazer compatibility provider is an optional application adapter. It
projects an evidence-qualified subset of HyperFlux controllers into the exact
D-Bus vocabulary expected by pinned OpenRazer 3.x clients. It is not the
receiver transport, a replacement for the official OpenRazer daemon, or the
preferred Polychromatic integration.

```mermaid
flowchart LR
    Client["OpenRazer-compatible client"]
    Provider["Optional compatibility provider"]
    SDK["Recovering HyperFlux SDK client"]
    Bridge["Bridge policy and transaction authority"]
    Client -->|"qualified D-Bus methods"| Provider
    Provider -->|"typed SDK intent"| SDK
    SDK --> Bridge
```

## Service Identities

| Mode | Bus name | Object root | Scope |
| --- | --- | --- | --- |
| Private default | `dev.hyperflux.OpenRazer1` | `/dev/hyperflux/OpenRazer1` | Explicitly HyperFlux-aware clients on the selected session bus |
| Legacy isolated | `org.razer` | `/org/razer` | One legacy client launched inside a new private `dbus-run-session` |

The package installs no D-Bus activation file. The default provider never owns
`org.razer`. Exact legacy identity is available only through
`hyperflux-openrazer-session -- <client command>`, which supervises the provider
and client in an isolated bus and removes both when the client exits. This keeps
the official daemon and unrelated direct devices untouched.

## Qualified Surface

An object is exported only when the bridge supplies an exact receiver profile,
exact child profile, qualified writable route, and the required lighting
capabilities. The adapter then exposes:

- exact pinned model name, image, kind, PID, and matrix dimensions;
- Static and Off lighting;
- whole-device brightness over a known complete frame;
- complete mouse per-LED and keyboard per-key Direct frames;
- software-effect frames generated by the client application.

Battery is intentionally absent. The pinned OpenRazer battery interface also
implies charging methods that HyperFlux has not qualified, so advertising only a
convincing fragment would mislead clients. DPI, polling rate, idle timeout,
pairing, firmware management, macros, scroll settings, and native hardware
effects are likewise absent until each receiver-side operation is independently
understood and qualified.

## Safety And Reconnection

Every write flows through the public SDK. The provider has no raw USB, HID, or
sysfs transport path and cannot turn a model name into write authority. Unknown,
unqualified, unavailable, or capability-incomplete devices are not exported.

Matrix rows are staged for one exact receiver generation and commit only as one
complete frame. A reconnect clears stale row buffers and retires old effect
leases. SDK recovery may reconnect reads, but an uncertain write result is never
replayed. If the bridge disappears, exported objects are removed and can return
only after a fresh qualified SDK view.

## Provenance And Verification

The method vocabulary and presentation metadata are bound to the OpenRazer
revision recorded in `integrations/catalog.json`. The adapter is a separate
`GPL-2.0-or-later` package with its own exact license payload. The
`openrazer-compatibility-contracts` verification node builds that wheel offline,
checks its entry points and license bytes, launches both identities in an
isolated D-Bus session, and drives the provider with the pinned upstream Python
client. These are software contracts; they do not claim new physical hardware
qualification.
